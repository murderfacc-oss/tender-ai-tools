# Таблица подбора оборудования

Что делать, когда пользователь говорит:
- «Сделай таблицу подбора оборудования»
- «Выгрузи характеристики»
- «Нужна таблица для поставщиков»
- «Сопоставь характеристики ЕИС и ТЗ»

---

## Где брать данные

**Источники в порядке приоритета (каскад):**

1. **ЕИС / описание объекта закупки** — табличные характеристики, если подача попозиционно. Главный источник.
2. **Техническое задание (ТЗ)** — функциональные и технические требования. Берём то, чего нет в табличных характеристиках ЕИС.
3. **Локальный сметный расчёт (ЛСР)** — конкретные позиции с товарными знаками. Сам по себе характеристик обычно не содержит — даёт марку/модель, по которой можно понять, что имел в виду заказчик.
4. **Интернет — поиск по марке/модели из ЛСР.** Применяется ТОЛЬКО когда характеристики позиции не описаны ни в ЕИС, ни в ТЗ, а в ЛСР есть конкретная модель. Брать официальный сайт производителя или крупный профильный источник (DNS, Citilink, профильные дистрибьюторы — Inforce, Сатро-Паладин и т.п.). **Обязательно сохранить URL** — он попадает в таблицу и в сводку в чате.

Если форма подачи **одной строкой** (нет табличных характеристик в ЕИС) — таблица составляется по ТЗ → ЛСР → интернет. Сообщить пользователю об этом и о том, что часть характеристик восстановлена из открытых источников.

❌ **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО** добавлять характеристики «из головы» / из тренировочных знаний модели. Если ни в документации, ни в интернете не нашёл — пометить `Не установлено` и сказать пользователю явно. Лучше пустая строка, чем выдуманное значение, под которое потом придётся подбирать оборудование.

---

## Обязательная атрибуция источников

Для **каждой** позиции таблицы обязательно указывать, откуда взяты характеристики. Это дисциплина против галлюцинаций и защита для пользователя при защите заявки и в случае спора с заказчиком.

**Источник указывается на двух уровнях:**

1. **В шапке позиции** — отдельная строка `Источник характеристик:` сразу перед списком характеристик.
2. **В сводке в конце таблицы (отдельным блоком в чате)** — короткая разбивка позиций по источникам, чтобы пользователь сразу видел, какие позиции «бумажные», а какие восстановлены из интернета и требуют дополнительной проверки.

**Допустимые метки источника:**

| Метка | Что обозначает |
|---|---|
| `ЕИС, таблица характеристик, поз. N` | Из табличных характеристик в извещении ЕИС. Главный источник. |
| `ТЗ, п. N` | Из текста технического задания, с конкретным пунктом. |
| `ЛСР, поз. N` | Из локального сметного расчёта (обычно только наименование/модель). |
| `Интернет: <URL>` | Найдено по марке/модели в интернете. Указать конкретный URL. Желательно официальный сайт производителя или крупный дистрибьютор. |
| `Не установлено` | Нет ни в документах, ни в интернете. Пометить явно — НЕ выдумывать. |

Если характеристики собраны из нескольких источников — перечислить все. Если в одной позиции часть характеристик из ЕИС, а часть с сайта производителя — указать оба источника и при необходимости привязать конкретные характеристики к источнику.

## Формат таблицы

**Главное:** удобно для работы с поставщиками. Каждая позиция — отдельный блок. Внутри позиции — характеристики **построчно**.

### Шаблон одной позиции

```
═══════════════════════════════════════════════════════════════
ПОЗИЦИЯ: <Наименование оборудования>
№ в ЕИС: <номер позиции, если применимо>
ОКПД 2: <код>
Кол-во: <количество> <ед. изм.>
Бренд в ЛСР (для эквивалента): <если есть товарный знак — указать>
Источник характеристик: <ЕИС, поз. N / ТЗ, п. N / ЛСР, поз. N /
                         Интернет: <URL> / Не установлено>

Характеристики:
  • <Характеристика 1>             <Значение / диапазон>
  • <Характеристика 2>             <Значение / диапазон>
  • <Характеристика N>             <Значение>

  ⚠ Несколько значений одной строкой (все обязательны):
    <Характеристика>: <значение 1>, <значение 2>, <значение 3>

Дополнительно из ТЗ (если применимо):
  • <Функциональные требования, не в табличных характеристиках>

═══════════════════════════════════════════════════════════════
```

### Пример (Воронеж СОШ № 70, позиция «Видеосервер»)

```
═══════════════════════════════════════════════════════════════
ПОЗИЦИЯ: Видеосервер
№ в ЕИС: 6
ОКПД 2: 26.20.16.110
Кол-во: 1 шт.
Бренд в ЛСР: Domination IP-80P-16-HS
Источник характеристик: ЕИС, таблица характеристик, поз. 6

Характеристики:
  • Количество видеоканалов            ≥ 80 шт.
  • Количество подключаемых дисков     ≥ 16 шт.
  • Минимальный объём подключаемого диска   ≥ 18 ТБ
  • Разрешение записи                  ≥ 12

  ⚠ Несколько значений одной строкой (все обязательны):
    Формат поддерживаемого видео: H.265, H.264, MJPEG

Дополнительно из ТЗ:
  • Круглосуточная запись со всех каналов с указанием
    номера камеры, даты и времени
  • Глубина архива не менее 30 суток (расчёт зависит от
    разрешения и битрейта камер)
  • Автоматическое восстановление записи после сбоя питания
  • Возможность подключения внешних батарей (резерв 60 мин.)

═══════════════════════════════════════════════════════════════
```

## КРИТИЧЕСКОЕ правило: «несколько значений одной строкой = все обязательны»

Когда в характеристиках в ЕИС значения разделены запятой / союзом «и» в **одной строке**, это означает: **оборудование должно соответствовать ВСЕМ перечисленным значениям**, не «одному из».

**Примеры из практики:**
- «Формат видео: H.265, H.264, MJPEG» — поддержка ВСЕХ трёх кодеков
- «Тип передачи данных: Ethernet» — единственное значение
- «Интерфейс подключения: Display Port, HDMI» — оба интерфейса должны быть

**В таблице это отражается:**
- Не разбивать одну характеристику на несколько строк (это потеряет логику «все»).
- Сохранять оригинальную строку с пометкой **⚠ (все обязательны)**.

**Если в инструкции к ЕИС используются разделители:**
- Запятая, союз «и» → все значения обязательны.
- Союз «или» → одно из значений на выбор.
- Знак `«/»` → требует уточнения (часто означает альтернативу).

Эту разницу отражать явно. Если непонятно — пометить «⚠ уточнить у заказчика, какой логический оператор подразумевается».

## Правила извлечения характеристик

### 1. Знаки сравнения сохранять точно

`≥`, `≤`, `<`, `>`, «не менее», «не более», «более», «менее», «до», «от»

Особо аккуратно с разницей:
- `<` (строго меньше, без равенства)
- `≤` (меньше или равно)
- «не более» = `≤`
- «менее» = `<`

Это **критическая ловушка** — типовая причина отклонения заявок.

### 2. Если значение фиксированное

«Тип камеры: Цифровая» — указать как есть, без вольностей.

### 3. Если есть «инструкция по заполнению заявки»

В разделе II «Инструкция» обычно есть фраза «Значение характеристики не может изменяться участником закупки» — это отдельная логика, отметить:

```
⚠ Значение фиксированное (не может изменяться участником):
  <Характеристика>: <Значение>
```

### 4. Единицы измерения

Указывать точно. «Ампер-час (3,6 кКл)» — так и оставлять, даже если громоздко.

## Что делать, если характеристики не описаны в документации

**Каскад действий:**

1. **Сначала** собери всё, что есть, по приоритету: ЕИС → ТЗ → ЛСР. Если хотя бы что-то нашлось — используй и пометь источник.
2. **Если в документации характеристик нет**, а в ЛСР есть конкретная марка/модель — **ищи в интернете** по этой модели. Использовать `WebSearch` / `WebFetch`. Брать официальный сайт производителя в первую очередь, затем крупных дистрибьюторов. **URL обязательно сохранить в источнике.**
3. **Если в интернете не нашлось** (модель снята с производства, у производителя нет сайта, открытые источники не дают характеристик) — пометь `Не установлено` и явно сообщи пользователю. Не пытаться восстановить «по памяти».

### Случай А — характеристики из ТЗ/ЛСР, без таблицы в ЕИС

```
═══════════════════════════════════════════════════════════════
ПОЗИЦИЯ: Кабель UTP с тросом
ОКПД 2: <если указан>
Кол-во: 622,2 м
Бренд в ЛСР: Caplex Cat5e 4 пары UTP PVC (внешний, -60/+60 °C),
             черный, коробка 305 м
Источник характеристик: ТЗ п. 4.2 + ЛСР, поз. 14
                        (таблица характеристик в ЕИС отсутствует —
                         форма подачи общая)

Характеристики (из ТЗ / ЛСР):
  • Витая пара UTP, 4 пары
  • Категория 5e
  • Внешнее исполнение (для уличной прокладки)
  • С тросом (для подвески)
  • Бухта 305 м

Критерий эквивалента: соответствие функциональным параметрам выше.

═══════════════════════════════════════════════════════════════
```

### Случай Б — характеристик в документации нет, есть только модель в ЛСР

Восстанавливаем характеристики из интернета по модели. Это допустимый и единственный корректный обход — модель из головы НЕ берём.

```
═══════════════════════════════════════════════════════════════
ПОЗИЦИЯ: Камера видеонаблюдения купольная
ОКПД 2: 26.30.50.119
Кол-во: 12 шт.
Бренд в ЛСР: Hikvision DS-2CD2T47G2-LSU/SL

Источник характеристик: Интернет — https://hikvision.ru/products/...
                        (характеристики в ЕИС/ТЗ не описаны;
                         восстановлены по модели из ЛСР через сайт
                         производителя)

Характеристики (по сайту производителя):
  • Матрица                       1/1.8" Progressive Scan CMOS
  • Разрешение                    4 МП (2688 × 1520)
  • Фокусное расстояние           2.8 / 4 мм
  • ИК-подсветка                  до 60 м
  • Форм-фактор                   Купольная
  • Степень защиты                IP67
  • Питание                       PoE 802.3af / 12 В DC

⚠ Характеристики восстановлены из открытого источника —
  заказчик в документации требований не описал. Перед подачей
  проверить, что именно эти параметры у конкретной партии товара
  актуальны (модель могла обновиться). Формальных критериев
  соответствия в закупке нет — это и плюс (свобода выбора), и риск
  (заказчик может постфактум сослаться на «то, что было в ЛСР»).
═══════════════════════════════════════════════════════════════
```

### Случай В — нет ни характеристик, ни модели

Если в ЛСР нет конкретной марки/модели (только обобщённое наименование вроде «коммутатор PoE 8 портов»), и в ЕИС/ТЗ характеристик тоже нет — искать в интернете не по чему. Пометить `Не установлено` и **спросить пользователя**, на какую конкретно модель он ориентируется, прежде чем продолжать.

```
═══════════════════════════════════════════════════════════════
ПОЗИЦИЯ: Коммутатор PoE
ОКПД 2: 26.30.11
Кол-во: 4 шт.
Бренд в ЛСР: не указан (общее наименование)

Источник характеристик: Не установлено
                        (в документации требований нет, в ЛСР нет
                         конкретной модели — нечего искать в сети)

⚠ Уточнить у пользователя: какую модель он рассматривает / какие
  у него критерии? Без этого нельзя корректно собрать строку
  таблицы — вписывать «типовые» характеристики из памяти модели
  ЗАПРЕЩЕНО.
═══════════════════════════════════════════════════════════════
```

## Сводная таблица в конце

После всех позиций — короткое резюме плюс **блок источников**. Этот блок отдельно дублируется в чате (не только в файле/таблице) — пользователь должен сразу видеть, какие позиции «бумажные», а какие восстановлены из интернета.

```
ИТОГО ПОЗИЦИЙ: <число>
  • С табличными характеристиками в ЕИС: <число>
  • С характеристиками из ТЗ / ЛСР:        <число>
  • Восстановлено из интернета по модели:  <число>  ⚠
  • Не установлено (нет ни в документах, ни в интернете): <число>  ⚠
  • С брендами в ЛСР (нужен эквивалент):   <число>
  • С пометкой «значение не может изменяться»: <число>

ИСТОЧНИКИ ПО ПОЗИЦИЯМ (для проверки перед подачей):
  Поз. 1.  Видеосервер              — ЕИС, поз. 6
  Поз. 2.  Камера купольная         — Интернет: https://hikvision.ru/...
  Поз. 3.  Кабель UTP с тросом      — ТЗ п. 4.2 + ЛСР поз. 14
  Поз. 4.  Коммутатор PoE           — Не установлено (нужно уточнить)
  ...

Позиции, помеченные ⚠ (восстановлены из интернета или не
установлены) — проверить вручную перед использованием в заявке.
Характеристики из открытых источников не являются требованиями
закупки и могут расходиться с фактической поставкой.
```

**В чат (сообщение пользователю)** после таблицы обязательно вывести именно блок ИСТОЧНИКИ ПО ПОЗИЦИЯМ — даже если сама таблица уехала в файл/Excel. Пользователь должен видеть, откуда что взято, без открытия файла.

## Опционально: выгрузка в Excel

Если пользователь говорит «выгрузи в Excel» / «нужен файл для отправки поставщику» — генерируешь Excel через `openpyxl`. Структура колонок:

| Позиция | № в ЕИС | ОКПД 2 | Кол-во | Ед. | Бренд в ЛСР | Характеристика | Значение | Все обязательны (если в одной строке) | Источник (ЕИС / ТЗ / ЛСР / URL / Не установлено) | Из ТЗ дополнительно |

Каждая характеристика — отдельная строка, объединение по позиции через объединённые ячейки в первой колонке. Колонка **Источник** заполняется обязательно для каждой строки. Если все характеристики позиции из одного источника — повторить значение или объединить ячейки в этой колонке.

Файл назвать: `equipment_table_<краткое имя закупки>.xlsx`. Положить в ту же папку проекта.

После генерации файла — в чате отдельным блоком вывести список «ИСТОЧНИКИ ПО ПОЗИЦИЯМ» (см. раздел «Сводная таблица в конце»).

## Чего НЕ делать

- ❌ **Не выдумывать характеристики из тренировочных знаний модели.** Это запрещено категорически — приводит к подаче заявки с ложными значениями. Каскад строго: ЕИС → ТЗ → ЛСР → интернет (с URL) → `Не установлено`. Никогда «по моим знаниям» / «обычно у таких камер» / «типовое значение для этого класса».
- ❌ **Не оставлять позицию без указания источника.** В шапке каждой позиции — строка `Источник характеристик:`. В конце таблицы — блок `ИСТОЧНИКИ ПО ПОЗИЦИЯМ`, продублированный в чате.
- ❌ **Не подменять источники.** Если часть характеристик из ЕИС, а часть с сайта производителя — указать оба источника, а не выбирать один «для красоты».
- ❌ **Не маскировать пустые места.** Если для позиции ничего не нашлось — пометить `Не установлено` и спросить пользователя. Не дописывать «правдоподобные» значения, чтобы строка выглядела заполненной.
- ❌ **Не разбивать «всё обязательно» на отдельные строки.** Это критическая ошибка.
- ❌ **Не «нормализовать» инженерные термины.** Если в ЕИС написано «Ампер-час (3,6 кКл)» — так и оставлять, не упрощать в «Ампер-час».
- ❌ **Не давать рекомендаций по конкретному оборудованию** («подайте Hikvision DS-2CD2T47G2»). Это работа подрядчика после получения таблицы.
- ❌ **Не сравнивать конкретные эквиваленты с оригиналом** автоматически — если только пользователь не попросил «сравни XVI EI8012CP1-D с Trassir TR-D2181IR3v3».
