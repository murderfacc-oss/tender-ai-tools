# Стартовый отчёт

Что выдаётся пользователю при первой загрузке документов закупки. Компактный отчёт по 8 пунктам, без лишних разделов.

---

## Структура отчёта

```
ЗАКУПКА: <одной строкой что закупается, заказчик, город>
Реестровый № <…> · Способ: <электронный аукцион / запрос котировок / иной>
НМЦК <сумма ₽> · Цена контракта <если торги прошли — сумма ₽, % снижения>

1. Контакты заказчика:
   <ФИО, должность>
   Тел.: <телефон>, <email>

2. Место работ:
   <адрес объекта; если несколько объектов — перечислить с номерами>

3. ОКПД 2: <код> — <наименование>

4. Форма подачи в ЕИС:
   <одной строкой / попозиционно / гибрид>
   <короткое следствие: что это значит для нацрежима / подачи заявки>

5. Структура описания:
   <ТЗ + ЛСР / Таблица характеристик в ЕИС / гибрид>
   <упоминание основных приложений: ВОР, схема, чертежи>

6. Проект: <не требуется / приложен / упомянут но не приложен>

7. Товарные знаки:
   <упоминаются ли в ЛСР / ТЗ — кратко с примерами>
   Оговорка «или эквивалент»: <ДА — цитата с указанием документа и пункта /
                                НЕТ — отдельный риск>

8. Требования к участнику:
   СРО — <требуется / не требуется + основание>
   Опыт по ст. 31 — <установлен / не установлен; если установлен — суммы и тип>
   Лицензии — <ФСБ / МЧС / иные / не установлены>
   <Иные нестандартные требования если есть>

Готов к разбору. Спрашивайте.
```

---

## Правила заполнения каждого пункта

### Заголовок

- Одной строкой: что закупается, ОПФ заказчика, город. Например: «Монтаж видеонаблюдения, МБДОУ детский сад № 44, Новороссийск».
- Реестровый номер закупки (из протоколов или из имени папки документации).
- Способ закупки — извлекать из проекта контракта или протоколов.
- НМЦК — из обоснования НМЦК или проекта контракта.
- Цена контракта — из протокола подведения итогов (если он есть в папке). Если нет — пометить «торги не прошли / в стадии / нет данных».

### 1. Контакты заказчика

- ФИО полностью (если есть).
- Должность.
- Телефон.
- Email.

Источник — реквизиты в проекте контракта (последний раздел) или в извещении.

Если контактов несколько (заказчик + контрактный управляющий) — перечислить обоих.

### 2. Место работ

- Полный адрес.
- Если **несколько объектов** — перечислить с номерами:
  ```
  2. Места работ (3 объекта):
     2.1. <адрес 1>
     2.2. <адрес 2>
     2.3. <адрес 3>
  ```

Источник — описание объекта закупки или п. «Место выполнения работ» в проекте контракта.

### 3. ОКПД 2

- Код + наименование.
- Если позиций несколько — перечислить главные:
  ```
  3. ОКПД 2:
     43.21.10.290 — Работы электромонтажные прочие (главный)
     26.40.33.111 — Камеры видеонаблюдения (для нацрежима)
  ```

### 4. Форма подачи в ЕИС

Это **критическая проверка**. Распознавать по разделу «Объект закупки» в требованиях к составу заявки и в описании объекта закупки.

**Одной строкой** (1 услуга / 1 работа / 1 усл. ед.):
```
4. Форма подачи в ЕИС: одной строкой («Выполнение работ по...», 1 усл. ед.)
   Нацрежим формально включён, но реестровые записи по позициям не требуются —
   облегчает подачу заявки.
```

**Попозиционно** (несколько типов оборудования с характеристиками):
```
4. Форма подачи в ЕИС: попозиционно (18 типов оборудования)
   На каждую позицию — свой ОКПД 2 и нацрежим. Реестровые записи требуются
   по позициям с применённым нацрежимом. См. п. 8 / отдельный запрос.
```

**Гибрид** (часть позиций расписана, часть — общими работами):
```
4. Форма подачи в ЕИС: гибрид
   Расписаны позиции: <…>
   Общими работами: <…>
   Нацрежим действует только на расписанные позиции.
```

### 5. Структура описания

Распознавать в документации:

- **ТЗ + ЛСР, без таблицы характеристик в ЕИС**: классическая «строительная» структура. ЛСР — основа объёма и цены.
- **Таблица характеристик в ЕИС**: оборудование описано через значения характеристик с диапазонами (≥, <, =) — типично для аукциона на товары/услуги.
- **Гибрид**: и то, и другое.

Упомянуть основные приложения:
```
5. Структура описания: ТЗ + ЛСР по Методике 2020 (РИМ).
   Приложения: ВОР, графики, схема монтажа.
   Таблицы характеристик в ЕИС нет.
```

или

```
5. Структура описания: таблица характеристик в ЕИС (18 позиций) + краткое ТЗ.
   ЛСР как обоснование НМЦК.
   Чертежей нет.
```

### 6. Проект

- **Не требуется** — если в описании объекта закупки прямо: «Государственная экспертиза не установлено», «Разрешение на строительство не требуется».
- **Приложен** — если в составе документации есть отдельные чертежи / схемы / РД.
- **Упомянут, но не приложен** — если в контракте или ТЗ ссылка на проект, а физически его в папке нет (это формальный пробел заказчика).

### 7. Товарные знаки и оговорка эквивалент

**Поиск товарных знаков** в ЛСР (наименования с конкретными артикулами / брендами): Dahua, Trassir, Hikvision, Bolid, ATIX, Cyfron, XVI, АйТек ПРО, и т.д.

Если товарных знаков **нет** — пометить «товарных знаков в ЛСР не выявлено». Это редкая ситуация для слаботочки.

Если товарные знаки **есть** — найти в документации **оговорку «или эквивалент»**. Точные места поиска:
- Описание объекта закупки (часто отдельный пункт)
- Требования к содержанию и составу заявки (раздел II «Инструкция по заполнению»)

Цитата с указанием документа и пункта:
```
7. Товарные знаки: упоминаются (Dahua DH-IPC-HFW2449SP, видеосервер
   Domination IP-80P, и др. в ЛСР).

   Оговорка «или эквивалент»: ДА.
   Описание объекта закупки, п. 7: «Во всех электронных документах
   и приложениях к извещению все товарные знаки читать как
   "товарный знак или эквивалент"».
```

Если оговорки **нет** — это **серьёзный риск** (заказчик может настаивать на конкретных моделях):
```
7. Товарные знаки: упоминаются (Trassir, и др.)
   Оговорка «или эквивалент»: НЕ НАЙДЕНА в документации.
   ⚠ Риск: заказчик может настаивать на конкретных брендах.
   Перед подачей — запрос разъяснения по эквиваленту.
```

### 8. Требования к участнику

**СРО НОСТРОЙ** — нужно или нет, основание.
- Часто требуется на капремонт ОКС (КВР 243).
- На монтаж слаботочки (КВР 244, ОКПД 33.xx или 43.21.10) обычно **не требуется**.
- Если требуется — указать конкретный пункт документации.

**Опыт по ст. 31**:
- ч. 1.1 — общий опыт.
- ч. 2 — доп. требования (через ПП № 2571).
- ч. 2.1 — особые требования.
- Если установлен — указать сумму и тип контрактов.

**Лицензии**:
- ФСБ (шифрование, гостайна).
- МЧС (пожарная безопасность).
- Иные отраслевые.

**Не упоминать** стандартные декларации по ст. 31 пп. 3-11 — это норма.

**Упомянуть**, если есть нестандартное:
- Сжатые сроки подачи (менее 7 рабочих дней).
- Доп. требования по ПП № 2571 (со ссылкой на номер позиции).

## Чего НЕ делать в стартовом отчёте

- ❌ Не выводить разделы «Что бросается в глаза», «Признаки сговора», «Прогноз поведения заказчика», «Слепые зоны», «Чек-листы действий». Это не часть стартового отчёта. На уточняющие вопросы отвечать в диалоге.
- ❌ Не давать оценок «беремся / не беремся».
- ❌ Не выдумывать факты. Если в документах не нашёл — пометить «в документах не нашёл».
- ❌ Не растягивать. Стартовый отчёт должен умещаться на одном экране.
- ❌ Не дублировать пустые пункты. Если по пункту нечего сказать — короткое «не установлено» / «не требуется» / «не выявлено».

---

## После отчёта — определение этапа закупки

После 8 пунктов **посмотри на структуру папки** и определи этап.

**Если в папке закупки есть подпапка `Контракт/`** — контракт уже подписан. Добавь в конец отчёта **короткую ремарку про этап исполнения / сдачи**:

```
📑 Контракт подписан. В папке Контракт/ — N документов из реестра контрактов:
   • <если есть подписанный контракт — упомянуть>
   • <если есть КС-2 / КС-3 — упомянуть, это значит работы сданы или сдавались>
   • <если есть документ о приёмке + платёжки — закупка фактически закрыта>

Этап работы: <анализ исполнения / подготовка к сдаче / сопровождение спора / закрытие>
По любым вопросам по подписанному контракту, исполнению или сдаче — спрашивай.
```

**Если подпапки `Контракт/` нет** — контракт ещё не подписан, добавь:

```
📋 Контракт пока не подписан. Этап работы: анализ закупки и подготовка к подаче / подписанию.
```

**Если есть `_changes.md`** (лог изменений от `update_tender`) — упомяни его в конце отчёта одной строкой:

```
ℹ В закупке были изменения после первоначальной загрузки — см. _changes.md.
```

## Завершение

Заканчивай отчёт строкой:

> Готов к разбору. Спрашивайте.

Это сигнал пользователю, что инициатива у него. Дальше — диалог по запросу.
