# scan-zakupki — Changelog

## v0.5 — 2026-05-06

**Чтение документов: native + pandoc вместо Python-парсинга.**

Раньше скилл инструктировал читать `.docx`/`.xlsx`/`.pdf` через
`python-docx`/`openpyxl`/`pdfplumber`/`easyocr`. Это:
- требовало ставить тяжёлые пакеты (`easyocr` тянет ~500 МБ моделей);
- теряло качество — pdfplumber путается на сложной вёрстке, на сканах
  не работает; python-docx не видит изображения и сложные таблицы.

Новый подход:

| Формат | Способ |
|---|---|
| PDF (включая сканы) | напрямую — модель мультимодальная, видит страницы |
| DOCX | `pandoc file.docx -o file.md`, дальше markdown |
| DOC | `soffice --convert-to docx`, потом pandoc |
| XLSX | `pandoc` или `soffice --convert-to csv` |
| Картинки | напрямую |

Python (`python-docx`/`openpyxl`/`pdfplumber`) — только fallback, когда
pandoc/soffice недоступны.

Также: добавлена прямая инструкция читать `print_form.txt` (если есть
в корне папки — это сводная карточка извещения с ЕИС, появляется
автоматически после `download_tender` v0.3+).

## v0.4 — 2026-05-05

**Распаковка архивов перед анализом.**

Добавлен шаг 0 в pipeline: до анализа скилл сканирует папку закупки на
наличие `.zip`/`.rar`/`.7z`/`.tar*` и распаковывает их в подпапку
`_unpacked/<имя_архива>/` рядом с оригиналом. Поддерживается:

- Рекурсивная распаковка (архив внутри архива, до 3 уровней).
- Fallback по кодировкам имён файлов (cp866 / cp1251 для старых русских архивов).
- Логирование результата одной строкой пользователю.

Закрывает кейс, когда документы закупки приходят как zip с zakupki.gov.ru
или присылаются коллегами в мессенджерах — раньше скилл их игнорировал.

## v0.3 — 2026-05-01

**Привязка к структуре MCP `zakupki-eis` + поддержка подписанного контракта.**

После доработки MCP `zakupki-eis` (новая функция `download_contract`, расширение `download_tender` с автоматическим скачиванием подписанного контракта в подпапку `Контракт/`) — обновили скилл, чтобы он:

- **Знал стандартную структуру папок MCP** (`1_ТЗ`, `2_Контракт` (проект), `3_НМЦК`, `4_Смета`, `5_Документация`, `6_Протоколы`, `7_Прочее`, `Контракт/` (подписанный), `_changes.md`).
- **Определял этап закупки по структуре**: есть ли подпапка `Контракт/` = подписан или нет; что внутри (КС-2/КС-3, платёжки, документ о приёмке) = сдача в процессе / завершена / расторгнута.
- **В стартовом отчёте после 8 пунктов** выводил короткую ремарку про этап (анализ vs исполнение vs закрытие).
- **На этапах исполнения и сдачи `lifecycle-guide.md`** имел триггеры через содержимое `Контракт/`, не только через слова пользователя.
- **При наличии `_changes.md`** прочитывал его первым, чтобы знать что в закупке менялось.

Не сломали обратную совместимость — на папках с произвольной структурой (как в калибровочных кейсах с подпапками `docs/`, `Закрытие/`, `Переписка/`) скилл продолжает работать.

## v0.2 — 2026-05-01

**Реархитектура: из «скилла-сценариста» в «помощника-коллегу».**

После обсуждения с пользователем переписали концепцию скилла. Старая модель (v0.1) делала большой `analysis.md` за один заход по 12-разделовому чек-листу. Это:
- Перегружало результат на нормальных закупках («лишняя информация», «переоценка рисков»)
- Не оставляло места для диалога — пользователь получал отчёт и закрывал
- Не сопровождало проект на этапах исполнения и сдачи

Новая модель (v0.2):
- При первой загрузке — компактный **стартовый отчёт** по 8 пунктам (см. `references/initial-report.md`)
- Дальше — **свободный диалог** с пользователем, без больших отчётов
- **Сопровождение проекта** на всех этапах: анализ → подача → подписание → исполнение → сдача → гарантия (см. `references/lifecycle-guide.md`)
- **Проактивные напоминания** на этапе исполнения: журналы, фотофиксация, акты скрытых работ, входной контроль материалов и т.д.
- **Таблица подбора оборудования** — отдельная команда (см. `references/equipment-table.md`), с правилом «несколько значений одной строкой = все обязательны»

### Что в скилле

```
scan-zakupki/
├── SKILL.md                              ← поведение помощника
├── CHANGELOG.md
└── references/
    ├── initial-report.md                 ← структура стартового отчёта (8 пунктов)
    ├── lifecycle-guide.md                ← подсказки на этапах закупки
    ├── equipment-table.md                ← формат таблицы подбора
    ├── legal-positions.md                ← правовые опоры (по запросу)
    └── responses-stroykontrolyu.md       ← шаблоны ответов стройконтролю (по запросу)
```

### Что удалено

- `references/checklist.md` — была 509-строчная процедура, заменена на справочные ремарки в `lifecycle-guide.md`
- `references/output-template.md` — нет больше единого формата вывода, диалог свободный

### 8 пунктов стартового отчёта

1. Контакты заказчика
2. Место выполнения работ (или несколько объектов)
3. ОКПД 2
4. Форма подачи в ЕИС (одной строкой / попозиционно / гибрид)
5. Структура описания (ТЗ + ЛСР / таблица характеристик / гибрид)
6. Наличие проекта
7. Товарные знаки + оговорка «или эквивалент»
8. Требования к участнику (СРО, опыт, лицензии)

### Этапы жизненного цикла

1. Анализ документации
2. Подача заявки
3. Подписание контракта
4. **Исполнение** (детальный этап с проактивными подсказками)
5. Сдача и закрытие
6. Гарантийный период
