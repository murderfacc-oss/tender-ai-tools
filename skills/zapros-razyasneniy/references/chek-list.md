# Чек-лист перед отправкой запроса

Финальная проверка перед генерацией Word-файла и подачей в ЕИС.

---

## Блок 1 — Стратегия

- [ ] Есть конкретная цель запроса (не «спросить для подстраховки»). См. [strategiya.md](strategiya.md).
- [ ] Знаю, что буду делать при ответе «да», при ответе «нет», при уклонении от ответа.
- [ ] Не превышен лимит запросов от одного лица (для электронного аукциона — 3 запроса).
- [ ] Если стратегическая цель — материал для жалобы, я готов её подать.

---

## Блок 2 — Срок подачи

- [ ] До окончания срока подачи заявок осталось не менее 4 дней (для электронного аукциона) либо 6 дней (для открытого конкурса).
- [ ] Запас минимум 1 день на случай технических сбоев в ЕИС.

**Если срок истекает в ближайшие 1–2 дня** — задавать поздно. Запрос либо не успеет получить ответ, либо вообще не будет принят.

---

## Блок 3 — Тип закупки

- [ ] Это **электронный аукцион** или **открытый конкурс** — разъяснения предусмотрены.
- [ ] Это **НЕ запрос котировок** (по ст. 50 44-ФЗ разъяснения не применяются).

Если запрос котировок — закрыть скилл, перенаправить на «протокол разногласий при подписании контракта».

---

## Блок 4 — Все вопросы упакованы в один документ

- [ ] **Все** вопросы пользователя по этой закупке собраны в **один** файл (это один запрос на счётчике лимита).
- [ ] Каждый вопрос оформлен отдельным пронумерованным блоком (1, 2, 3...).
- [ ] Между блоками нет повторов и противоречий.
- [ ] Слабые / опасные вопросы отбракованы (см. [strategiya.md](strategiya.md)).

---

## Блок 5 — Текст запроса

- [ ] Цитата из документации **точно** соответствует тексту в извещении (проверено по оригиналу).
- [ ] Указан конкретный документ: «Техническое задание», «Локальный сметный расчёт», «Описание объекта закупки», и т.д. — с номером Приложения.
- [ ] Указан конкретный пункт или позиция (например, «п. 5.2», «позиция № 14»).
- [ ] Если используются числа — они есть с единицами измерения («32 ТБ», «18 штук», «200 метров»).

---

## Блок 6 — Человеческий язык

- [ ] Прочитал текст вслух — каждая фраза понятна без специальных знаний.
- [ ] Заменил все технические термины (WDR, PoE, lux, битрейт, FPS) на обывательские. См. [chelovecheskiy-yazyk.md](chelovecheskiy-yazyk.md).
- [ ] Минимум юридического языка. Используется только если без него никак (например, в сюжете 5 — со ссылкой на ст. 33 44-ФЗ).
- [ ] Тон дружелюбный («прошу подтвердить», «прошу уточнить», «этот вопрос задан для…»). Никаких «требуем», «грубо нарушает».

---

## Блок 7 — Закрытый формат ответа

- [ ] Вопрос сформулирован так, что заказчику легко ответить «да» / «нет», «вариант А» / «вариант Б», или назвать конкретное число.
- [ ] Есть варианты ответа в самой формулировке (когда применимо).
- [ ] Нет открытых вопросов «расскажите подробно», «дайте разъяснение по всем требованиям».

---

## Блок 8 — Формальные элементы документа

В файле **только**:

- [ ] **Заголовок:** «Запрос на разъяснение положений извещения о проведении закупки № [рег.№]»
- [ ] **Вступительная фраза:** «Уважаемые специалисты! Прошу разъяснить положения извещения по следующим вопросам.»
- [ ] **Тело:** пронумерованные блоки вопросов.
- [ ] **Подпись:** «С уважением, [ФИО]».
- [ ] **Дата:** актуальная дата составления.
- [ ] Имя файла: `Запрос разъяснений по закупке № [рег.№].docx`.

**В файле НЕ должно быть:**
- ❌ Шапки с адресатом (наименование заказчика, ИНН, адрес)
- ❌ Реквизитов заявителя (наименование, ИНН, адрес, телефон, e-mail)

Реквизиты обеих сторон автоматически прикрепляются ЕИС при подаче запроса. Если их добавить в файл — это ничего не сломает, но визуально засоряет документ.

---

## Блок 9 — Технические требования к файлу

- [ ] Формат: **.docx**.
- [ ] Шрифт: Times New Roman, 12 pt.
- [ ] Размер страницы: **А4**.
- [ ] Поля: стандартные (2 см сверху/снизу, 3 см слева, 1.5 см справа).
- [ ] Документ открывается без ошибок (тестовое открытие в любом текстовом редакторе).

Технические требования автоматически выдерживает скрипт [generate_request.py](generate_request.py).

---

## Блок 10 — После генерации Word-файла

- [ ] Пользователь открыл файл и прочитал его (последняя проверка вручную).
- [ ] Файл подписан УКЭП (на уровне ЭП — не в самом Word).
- [ ] Файл загружен в ЕИС в раздел «Запросы на разъяснения».

---

## Финальное сообщение пользователю

После прохождения всех блоков скилл выдаёт пользователю:

```
Запрос разъяснений готов.

📄 Файл: Запрос разъяснений по закупке № [рег.№].docx
   Путь: [полный путь к файлу]

✅ В документе [N] вопросов: [сюжет 1] / [сюжет 2] / ...
✅ Все вопросы упакованы в один файл (списывается с лимита один раз)
✅ Закрытые форматы ответов
✅ Технические требования к файлу: TNR 12, А4
✅ Реквизиты сторон в файле НЕ указаны — их прикрепит ЕИС
✅ Срок подачи в ЕИС: до [дата] [время]

Дальнейшие действия:
1. Откройте файл и прочитайте текст ещё раз — это последняя проверка.
2. Если всё в порядке — подпишите УКЭП и загрузите в ЕИС
   в раздел «Запросы на разъяснения».
3. После получения ответа от заказчика (срок: до 2 рабочих дней) —
   сохраните распечатку ответа из ЕИС.
   Это защитный документ при приёмке работ.

Прогноз: ответ заказчика поступит до [дата].
По практике — [высокий / средний / низкий] шанс получить нужный нам ответ.
```

Если хоть один пункт чек-листа не пройден — сообщить пользователю **точно какой**, и предложить либо доработать запрос, либо подать «как есть» с предупреждением о рисках.
